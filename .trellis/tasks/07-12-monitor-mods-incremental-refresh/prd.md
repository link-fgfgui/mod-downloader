# 监控 mods 目录并增量刷新本地 Mod

## 目标

让本地 Mod 管理在大多数情况下无需用户手动点击刷新：当当前实例的 `mods` 目录确实发生变化时自动更新界面；启用/禁用单个 Mod 不再触发整个目录的昂贵全量扫描。

## 背景

- `Manage.vue` 在页面激活时和手动刷新时调用 `refreshSelectedMods()`。
- `ApplyLocalModBatchOperation` 重命名后调用 `RefreshSelectedVersionMods`，现有流程会清空实例索引并扫描全部 JAR。
- `ScanModsDir` 对每个 JAR 计算 SHA1/CurseForge 指纹并解析元数据；没有运行时 `mods` 目录 watcher。

## 需求

1. 使用 Go `fsnotify` 监听当前选中实例的 `mods` 目录；随选中实例、Minecraft 目录变化和服务关闭正确启停。
2. 监听新增、删除、重命名、启用/禁用（`.jar` 与 `.jar.disabled`）和写入完成事件，使用短暂 debounce 合并重复事件。
3. 单文件变化只计算 hash、解析和更新受影响 JAR；删除/重命名只移除或更新受影响路径，不重扫其他 JAR。
4. 启用/禁用/删除操作复用增量更新路径，避免同步等待全量扫描。
5. 自动刷新通过现有 `selected-version-changed` 事件更新 Manage 页面。
6. 监听异常、无法判定影响范围或增量失败时回退全量刷新并记录日志；手动刷新仍保留为全量兜底。
7. 保持下载、批量操作、切换实例兼容，不让旧实例事件污染当前实例。

## 验收标准

- 外部新增、删除或重命名当前实例 JAR 后，页面在合理延迟内自动更新。
- 单 Mod 启用/禁用耗时不再随其他 JAR 总大小线性增长。
- 未变化 JAR 不重新计算 hash 或解析内容。
- watcher 在切换实例后只作用于新实例，服务关闭后释放。
- 重复事件不会造成刷新风暴；失效时手动刷新可恢复正确状态。
- 现有 Go、本地 Mod 操作和前端构建测试继续通过。

## 范围外

- 不改变 Mod 元数据格式、在线平台协议或展示字段。
- 不要求跨应用实例同步，不移除手动刷新按钮。
