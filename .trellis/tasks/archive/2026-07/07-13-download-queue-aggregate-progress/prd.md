# Download queue aggregate progress

## Goal

让用户在下载队列中同时掌握整个队列的完成程度、每个下载任务的完成程度，以及当前队列的总下载速度，从而判断整体下载状态并定位停滞任务。

## Background

- 下载队列由 `core/downloader` 持有，`DownloadQueueState` 经 `appcore`、Wails 的 `download-queue-updated` 事件和 Pinia store 投影到 `frontend/src/App.vue`；该功能必须保持这一单一数据源边界（`.trellis/spec/guides/cross-layer-thinking-guide.md`）。
- 当前队列支持多个并行运行任务，并将运行中、等待中、失败和取消任务投影为 `DownloadQueueItem`，但状态结构不含字节进度或速度（`core/structs/search.go:47`、`core/downloader/download.go:1224`）。
- 文件传输层已有线程安全的 `ProgressTracker`，可提供 `BytesComplete` 与 `TotalBytes`；下载器当前每秒读取一次已完成字节来检测停滞，但没有把采样值写入队列状态（`core/downloader/filetransfer/types.go:36`、`core/downloader/download.go:1353`）。
- Modrinth 与 CurseForge SDK 的版本文件结构都提供文件大小，但当前统一 `ModVersion` 及 provider 映射没有保留该字段（`core/models/models.go:23`、`core/providers/modprovider.go:946`、`core/providers/modprovider.go:1083`）。
- 成功任务当前会立即从 `DownloadQueueState.Items` 消失；失败和取消任务保留在有界、仅内存的重试列表中（`core/downloader/download.go:516`、`core/downloader/download.go:1224`）。
- 前端队列是一个 420px 宽的菜单面板，并具有离场快照；新增动态指标必须一并进入快照，避免队列结束时可见内容在动画完成前清空（`frontend/src/App.vue:22`、`frontend/src/App.vue:240`、`.trellis/spec/app/frontend/ui-state-lifecycle.md`）。

## Requirements

- R1. 下载队列状态必须提供每个等待中或运行中任务的已下载字节数与总字节数；等待任务在已有 provider 文件大小时从 0 开始，运行任务以传输层探测值为准实时更新。
- R2. 每个等待中或运行中任务必须显示独立进度条。总大小有效时显示确定进度；总大小未知时显示不确定进度。失败和取消的重试记录不显示活动下载进度。
- R3. 队列必须显示按真实字节数加权的总进度，计算口径为当前队列周期内已知大小任务的累计已下载字节除以累计总字节，不按任务数等权平均。
- R4. 队列周期从空闲状态首次入队开始，到所有等待中和运行中任务结束为止。等待任务计入总量，成功任务的完成字节保留到周期结束，任务仅在开始或完成时不得导致总进度回退。
- R5. 总大小未知的任务不参与总进度分子和分母；取得有效总大小后再纳入聚合。动态发现新依赖、未知大小变为已知或重试重置传输时，因队列工作范围变化允许总百分比调整。
- R6. 失败或取消任务离开活动队列时不再参与当前周期聚合；重试后以新活动任务重新计入。仅剩失败或取消记录时不显示活动总进度与速度。
- R7. 下载队列必须显示所有当前运行任务的总下载速度。速度以传输字节增量按约一秒采样并聚合，空闲或无字节增长时为零。
- R8. 总进度、总速度和任务进度必须随现有 `download-queue-updated` 状态事件更新，不新增第二套前端计时或进度所有者。
- R9. 新增展示必须适配现有紧凑队列面板、中文和英文文案，并保持队列离场快照、取消、运行中重试、失败重试、动态依赖、完成通知和并发下载行为不变。

## Acceptance Criteria

- [x] AC1 (R1, R2): 已知总大小的等待任务显示 0% 确定进度；运行时任务进度随已下载字节增加并最终达到 100%。
- [x] AC2 (R2, R5): 总大小未知的活动任务显示不确定进度，并且不会污染已知任务的队列总百分比。
- [x] AC3 (R3, R4): 两个大小不同的任务按字节而非任务数加权；一个任务完成并从可见活动项移除后，其完成字节仍保留到其他活动任务结束。
- [x] AC4 (R4, R5): 普通任务开始、并发切换和成功完成不会使总进度回退；新增动态依赖、大小由未知变为已知或重试重置时可以按新的总工作量重新计算。
- [x] AC5 (R6): 取消或失败任务不阻止剩余活动队列达到 100%；仅有重试记录时隐藏活动总进度与速度，重新重试后开始新的活动统计。
- [x] AC6 (R7): 多个并行任务下载时，总速度等于各运行任务最近采样速度之和；停滞、完成、取消或队列空闲后速度归零。
- [x] AC7 (R8, R9): 队列事件可驱动总进度、总速度和任务进度刷新，无需重新打开面板或手动刷新。
- [x] AC8 (R9): 新指标在现有面板宽度和窄屏约束下不与标题、状态或操作按钮重叠，并具有中英文显示。
- [x] AC9 (R9): 取消、重试、动态依赖排序、SHA1 重试、停滞重试、完成声音和队列离场快照的既有测试或验证仍通过。
- [x] AC10: `core` 与应用 Go 测试、构建和 vet，以及前端 lint 和生产构建全部通过；涉及共享结构后 Wails 前端类型已重新生成。

## Out Of Scope

- 改变下载并发数、分片并发、自适应并发、重试策略或下载调度顺序。
- 显示单任务下载速度、预计剩余时间、长期速度图表或跨应用重启的下载历史。
- 为未知大小任务额外发起独立网络探测；继续使用 provider 元数据和实际传输探测结果。
- 将成功任务作为历史记录长期保留在队列面板中。
